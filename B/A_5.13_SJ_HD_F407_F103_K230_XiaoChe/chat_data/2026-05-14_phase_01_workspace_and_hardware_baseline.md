# 阶段摘要 01：工作区建档与硬件关系基线

- 日期：2026-05-14
- 阶段编号：`01`
- 主题：`workspace_and_hardware_baseline`
- 范围：`workspace root / F103 / F407 / OpenMV / K230`

## 1. 本阶段目标

先把整个项目放到一个可持续推进的框架里，再去做具体代码工作。
这一阶段的目标不是直接写功能，而是先完成两件基础工作：
- 建立根目录下的工程管理区
- 结合现有工程、原理图和网表，确认系统真实结构

## 2. 本阶段形成的目录基线

用户一开始就明确提出，要把管理目录放在根目录下，并且后续改成英文名，避免中文路径给工具链、脚本和编译环境带来兼容性问题。

最终建立的目录为：
- `dev_logs`
- `chat_data`
- `backups`
- `temp_transfer`
- `reference_library`

这一步的意义在于，后续所有日志、对话沉淀、备份、中转文件和资料库都有明确落点，工程管理从一开始就被纳入正式流程。

## 3. 本阶段完成的工作区扫描

扫描根目录后，项目被识别为由四个主要部分组成：
- `F103 grayscale sensor(HD)`
- `F407 main control panel`
- `OpenMV Visual module(SJ)`
- `K230 Visual module(SJ)`

其中当时的实际情况是：
- `F103` 与 `F407` 是现阶段已有实质内容的主体
- `OpenMV` 为最终视觉主方案预留目录
- `K230` 为备选视觉方案预留目录

## 4. 本阶段完成的关键澄清

这一阶段最重要的价值，不是“看见了哪些文件”，而是把系统关系从容易误判的状态，澄清成了较稳定的工程认识。

经过用户补充和后续原理图、网表交叉确认，形成了以下关键结论：

1. `F103` 是主用灰度模块，不是临时测试板。
2. `F407` 上的 `灰度-IN` 对应另一块“无 MCU 的 8 路灰度板”，并不是 `F103`。
3. 最终视觉链路准备接 `OpenMV`，`K230` 作为备选。

## 5. 本阶段形成的硬件基线认识

围绕 `F103`，本阶段已经建立了后续可直接复用的硬件基线：
- 主控芯片：`STM32F103C8T6`
- 灰度前端：`8x ITR20001/T`
- 模拟处理：`2x LM324`
- 阈值调节：`8x 可调电位器`
- 对外接口：`I2C + UART`

同时还逐步确认了：
- `CN2` 为 `3V3 / GND / PB10 / PB11`，对应 `I2C`
- `CN3` 为 `3V3 / GND / OUT_RX / OUT_TX`
- 网名是“对端视角”命名：
  - `OUT_RX` 实际连接 `F103 USART1_TX`
  - `OUT_TX` 实际连接 `F103 USART1_RX`

这为后面决定“F103 与 F407 先走串口联调”打下了直接基础。

## 7. 这一阶段为什么重要

如果没有这一阶段，后续协议设计和代码接入很容易建立在错误前提上。

这一阶段真正完成的是：
- 把工作区从“散文件集合”变成“可管理工程”
- 把系统从“凭目录猜结构”变成“由原理图和网表支撑的硬件认识”
- 把 `F103 / F407 / OpenMV / K230` 的角色关系先固定下来

## 8. 本阶段之后最自然的下一步

完成这一阶段之后，最自然的后续推进方向就变成了：
1. 基于已确认的接口关系设计 `F103 <-> F407` 的通信方案
2. 先聚焦 `F103` 单板，跑通灰度采样、LED 指示和串口输出
3. 再逐步把调试协议收敛为适合运行态的短帧协议

也正是基于这个基础，项目随后进入了 `phase_02` 所记录的 `F103` 运行态短帧阶段。

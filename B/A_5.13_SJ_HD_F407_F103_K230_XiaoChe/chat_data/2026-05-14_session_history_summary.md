## 4. 关键硬件关系的确认过程

本轮会话中，围绕 `F103` 和 `F407` 的关系，发生了几次非常重要的澄清。

用户明确确认了三件事：

1. `F103` 是主用灰度模块。
2. `F407` 上的 `灰度-IN` 对应另一块“无 MCU 的 8 路灰度板”，不是 `F103`。
3. 最终视觉方案准备接 `OpenMV`，`K230` 只是备选。

结合原理图和 `.enet` 网表，最终形成的稳定理解是：

- `F103` 不是简单测试板，而是一块带本地处理能力的智能灰度模块。
- `F103` 板上包含：
  - `8x ITR20001/T`
  - `2x LM324`
  - `8x 可调电位器`
  - `STM32F103C8T6`
  - 对外 `I2C` 与 `UART` 接口
- `F407` 同时兼容两种灰度方案：
  - 智能灰度模块 `F103`
  - 无 MCU 的 8 路原始灰度板

## 5. 已形成的接口理解

### 5.1 F103 对外接口

- `CN2`：`3V3 / GND / PB10 / PB11`，对应 `I2C`
- `CN3`：`3V3 / GND / OUT_RX / OUT_TX`

经网表确认：

- `OUT_RX` 实际连 `F103 USART1_TX`
- `OUT_TX` 实际连 `F103 USART1_RX`

### 5.2 F407 对应接口

- `灰度-i2c-IN`：对应智能灰度模块的 I2C 输入
- `灰度-串口-IN`：对应智能灰度模块的 UART 输入
- `灰度-IN`：对应另一块无 MCU 的直接 8 路灰度板
- `视觉信号out`：对应 `UART4`，用于 `OpenMV`
- `视觉供电out`：提供 `12V + GND`

## 9. F103 串口优先调试版本的形成

为适应这个目标，`F103` 工程进一步增加了：

- `Core/Inc/gray_app.h`
- `Core/Src/gray_app.c`

这一阶段完成了以下事情：

1. 建立 `gray_app` 轮询任务。
2. 读取 8 路真实灰度输入：
   - `PA2`
   - `PA3`
   - `PA4`
   - `PA5`
   - `PB12`
   - `PB13`
   - `PB15`
   - `PA8`
3. 把 8 路状态映射到板载 `D1~D8` 指示灯。
4. 通过 `USART1` 周期发 `GRAY_STATUS` 长帧。
5. 使用 `KEY1` 切换连续发送开关。
6. 使用 `PB14(KEY1-OUT)` 做发送状态指示。

## 10. 实际串口抓包验证节点

在单板串口调试阶段，用户提供了一大段 `HEX` 抓包。

通过解帧得到的核心结论是：

- 帧头稳定为 `AA 55`
- `SRC = 0x20` 正确表示 `F103`
- `TYPE = 0x10` 正确表示 `GRAY_STATUS`
- `LEN = 0x0008`
- CRC 正常
- 帧序号与帧计数连续增长
- `sensor_bits`、`track_type`、`center_err` 会随遮挡与路型变化而变化

由此确认：

- 串口链路已经跑通
- 打包逻辑没乱
- 语义方向开始具备可分析性

## 11. 指示灯亮灭反向问题

在实际调试中，用户又发现：

- 板载指示灯的亮灭与期望显示方向相反

因此代码做了一个重要收口：

- 将“传感器极性”和“LED 显示极性”拆开处理

具体变成两个独立宏：

- `GRAY_SENSOR_ACTIVE_IS_LOW`
- `GRAY_LED_ACTIVE_IS_LOW`

这样做的意义是：

- 不必为了修正 LED 显示，立刻翻转串口语义
- 可以先让板上显示符合人的直观观察
- 串口 `sensor_bits` 语义则继续单独验证

## 12. 对运行态协议的新思路

后续对话里，用户进一步提出新的系统目标：

- 不希望主控再去解析太多原始灰度数据
- 尽可能让 `F103` 在本地完成理解和判断
- 发给主控的应该更接近“应该左拐还是右拐，应该拐多少”

围绕这一点，对话形成了一个关键共识：

- 对运行态而言，“只发 8 路原始灰度位图”并不是最优方案
- 更合理的是：
  - 让 `F103` 本地做更多理解
  - 主控只接收精简后的控制量和场景判断

## 13. 运行态短帧的正式提案

在此基础上，形成了运行态短帧提案：

```text
AA 55 TURN SCENE FLAGS CRC8
```

总长 `6` 字节。

这份提案的核心设计是：

- `TURN`
  - 直接表示转向建议量
  - 负左正右
  - 绝对值表示修正强度
- `SCENE`
  - 表示当前最主要的路段场景
- `FLAGS`
  - 补充表达可靠性、附加特征与辅助状态

当前提案中重点考虑过的场景包括：

- 普通直行
- 普通左转
- 普通右转
- 左直角
- 右直角
- 十字
- 左支路
- 右支路
- T 字
- 停车线 / 大面积黑区
- 丢线
- 模糊态

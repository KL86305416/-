# 阶段摘要 10：F407 按键点灯最小自检
- 日期：2026-05-14
- 阶段编号：`10`
- 主题：`f407_key_led_selftest`
- 范围：`F407 main control panel / backups / dev_logs / chat_data`

## 1. 本阶段触发背景

在确认之前的下载故障来自下载器损坏之后，用户提出了一个非常务实的下一步：

- 先别急着上整车控制
- 先确认 `F407` 主控自己到底是不是正常活着

为此，本阶段选择了最基础也最可靠的一组信号做板级自检：

- 读 `D5 / D6` 按键
- 控 `E0 / B9` 指示灯

## 2. 本阶段做出的关键判断

这一步不是随便写个 `if`，而是先把引脚和电气方向坐实：

- `D5 = PD5 = KEY_3`
- `D6 = PD6 = KEY_4`
- `B9 = PB9 = LED_1`
- `E0 = PE0 = LED_2`

同时通过网表确认：

- 按键是 `10K` 上拉、按下接地
- 所以按键是低电平触发
- 两个 LED 都是 GPIO 拉高点亮

这保证了本轮自检不会因为极性搞反而误判主控状态。

## 3. 本阶段真正落地的行为

本轮没有拆掉原有主控框架，而是在现有程序上叠加了一个最小自检逻辑：

- 上电默认两灯熄灭
- 轮询 `PD5 / PD6`
- 任一按键第一次按下
- `PB9 / PE0` 两灯同时点亮并锁存保持

之所以做成“按一下后保持亮”，而不是“按住才亮”，是为了让板级验证更直观：

- 更容易看出主控有没有真正检测到按键事件
- 更适合做“板子活没活”的肉眼确认

## 4. 本阶段对工程状态的意义

这一小步的价值其实很大，因为它提供了一个非常低门槛的板级判断标准：

只要烧录后能出现：

- 按下 `D5` 或 `D6`
- `E0 / B9` 亮起并保持

就至少说明：

- 程序已经真的跑起来了
- 主循环在执行
- 输入口是好的
- 输出口是好的
- 当前固件确实进了我们写的新逻辑

这比一上来就拿整套寻迹逻辑判断主控生死，要稳得多。

## 6. 下一步最自然的动作

本阶段之后最自然的下一步，就是把这版固件烧到 `F407` 上做实板验证：

1. 看上电时两灯是否默认熄灭
2. 按 `D5`
3. 按 `D6`
4. 观察 `E0 / B9` 是否点亮并保持

如果这一步成立，就说明主控的最小生命体征已经确认，可以继续往下推进更复杂的：

- `F103 -> F407` 数据联调
- OLED 灰度页观察
- 再到电机控制接入

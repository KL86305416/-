# 阶段摘要 04：F103 板载 LED 左右方向修正

- 日期：2026-05-14
- 阶段编号：`04`
- 主题：`f103_led_left_right_fix`
- 范围：`F103 grayscale sensor(HD) / chat_data / dev_logs`

## 1. 本阶段目标

根据最新上板观察，修正 `F103` 板载 `D1 ~ D8` 的左右显示方向，让 LED 可视化与实际车头左右方向一致。

这一阶段强调的是：
- 修正板上观察体验
- 不连带改动串口位序
- 不破坏已经确定下来的运行态短帧语义

## 2. 进入本阶段前的状态

在进入本阶段前，代码里采用的是最直接的镜像方式：
- `bit0 -> D1`
- `bit1 -> D2`
- ...
- `bit7 -> D8`

这在“逻辑编号顺序”和“物理排布顺序”一致时没有问题，但上板观察表明，板上 LED 的左右物理顺序与最初假设相反，因此显示出来会呈现左右镜像。

## 3. 本阶段的关键决策

本阶段没有去动：
- `sensor_bits` 的位定义
- `TURN` 的左右符号
- `SCENE / FLAGS` 的语义

而是选择了更稳妥的修正方式：
- 只在 `GrayApp_UpdateSensorLeds()` 内部，把 LED 显示做左右镜像

也就是说，修的只是“人眼看到的板载显示”，不是协议本体。

## 4. 本阶段代码层面的落地

本阶段在 `gray_app.c` 中新增了一个独立控制宏：
- `GRAY_LED_MIRROR_LEFT_RIGHT`

当前设置为：
- `1U`

实现效果是：
- `D1` 显示 `IN8 / bit7`
- `D2` 显示 `IN7 / bit6`
- ...
- `D8` 显示 `IN1 / bit0`

这样做之后，板载 LED 的左右视觉顺序就和实际板上方向对齐了。

## 5. 为什么这一步要和协议语义分开

如果这一步直接去改：
- 传感器读入顺序
- `center_err` 位置表
- `left/right` 场景判定

那就不只是“修指示灯”，而是会把整个运行态短帧的左右语义一起翻掉。

当前我们还处在 `F103` 单板联调阶段，这时候最稳的办法，是先把：
- 板上可视化
- 串口语义
- 运行态决策

三者分开控制，逐个校正。

## 6. 本阶段同步更新的文档

为了避免文档与代码脱节，本阶段还同步更新了：
- `dev_logs/2026-05-14_f103_uart_first_debug.md`

把 `D1 ~ D8` 的当前显示内容改成了镜像后的实际版本，并补充说明这是“仅修 LED 左右方向、不改协议语义”的修正。

## 7. 当前结论

截至本阶段，已经可以明确：

- LED 亮灭极性和左右方向，已经被拆成两个独立维度处理
- 当前这次修正只处理了“左右方向”
- `sensor_bits / TURN / SCENE / FLAGS` 的定义保持不变

这让后面继续验证“左负右正”“位序是否与车头一致”时，不会被板载显示问题干扰。

## 8. 本阶段之后最自然的下一步

完成这一步之后，最自然的下一步是重新上板确认两件事：

1. 板载 `D1 ~ D8` 的左右显示是否已经符合实际
2. 串口短帧里的左右语义是否依旧符合“左负右正”

如果 LED 方向已经正确，但串口方向仍反，那下一步就该进入更深一层的“传感器位序 / 位置权重 / 左右语义”校正，而不再是 LED 显示问题本身。

# 阶段摘要 13：F407 TB6612 分支接回 USART6 短帧与 OLED 监视
- 日期：2026-05-15
- 阶段编号：`13`
- 主题：`f407_tb6612_uart_oled_migration`
- 范围：`F407 main control panel TB6612 / backups / dev_logs / chat_data`

## 1. 触发背景

在上一阶段确认 `TB6612` 分支仍是硬件骨架之后，本阶段开始做第一步真正的代码迁移：

- 不急着碰电机闭环
- 先把 `F103 -> F407` 运行态数据链路接回来
- 先让主控重新具备“看见灰度判断结果”的能力

这一步的目标，是先把主控重新拉回可观察、可联调的状态。

## 2. 本阶段实际完成的事

本阶段完成了三件核心工作：

1. 把 `gray_module` 迁入 `TB6612` 分支
2. 把 `oled_display` 和 `gray_monitor` 迁入 `TB6612` 分支
3. 改写 `main.c`，让 `USART6` 中断接收和 OLED 轮询真正跑起来

同时，Keil 工程文件也已经同步挂上这些新源文件，不是只把文件扔进目录里。

## 3. 当前主控已经具备的能力

现在这套 `TB6612` 分支已经可以做这些事：

- 通过 `USART6` 按字节接收 `F103` 的运行态短帧
- 本地做 `AA 55 + turn + scene + flags + crc8` 校验与解析
- 在 `I2C OLED` 上显示：
  - 在线状态
  - turn
  - scene
  - flags
  - 若干关键状态位

也就是说，这条分支现在第一次具备了“主控侧重新观察 103 输出结论”的能力。

## 4. 本阶段的意义

这一步虽然还没让车动起来，但意义非常大：

- `TB6612` 分支不再只是空外设工程
- 它已经重新接回了主控运行态最关键的输入观察窗口

这会直接降低后面电机适配时的调试难度，因为我们不需要一边怀疑串口、一边怀疑 OLED、一边怀疑电机，只要先确认数据链路正常，再单独推进执行层即可。

## 5. 当前仍保留的边界

本阶段有意没有继续往下做：

- 没改 `PA15 / DJ_BIN2`
- 没改 `TIM1 / TIM2` PWM 频率
- 没写 `TB6612` 版 `motor_drive`

这样可以保持本阶段结论单纯：

- 如果现在上板看到 OLED 正常刷出 `F103` 数据，就说明通信和显示链路已经被成功迁回来了

## 6. 下一步最自然的动作

下一步最自然的推进顺序已经非常明确：

1. 上板确认 `USART6 + OLED` 现象是否正常
2. 然后开始修 `PA15` 与 PWM 基础参数
3. 最后接 `TB6612` 电机驱动层

这会继续延续当前项目一直在用的节奏：

- 先恢复观测
- 再恢复执行
- 最后进入整车联调

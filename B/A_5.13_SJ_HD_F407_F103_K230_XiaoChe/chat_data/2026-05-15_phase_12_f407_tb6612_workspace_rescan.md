# 阶段摘要 12：F407 TB6612 分支重扫描与结构判断
- 日期：2026-05-15
- 阶段编号：`12`
- 主题：`f407_tb6612_workspace_rescan`
- 范围：`workspace root / F407 main control panel TB6612 / dev_logs / chat_data`

## 1. 触发背景

本阶段的目标不是立刻改代码，而是先重新扫描整个工作区，并重点分析：

- `F407 main control panel TB6612`
- 其代码骨架
- `.ioc` 引脚定义
- 原理图网表与接口关系

这样做的目的，是先把这条新主控分支的真实起点摸清楚，避免把旧分支的理解直接套错到新工程上。

## 2. 本阶段得到的关键判断

本阶段确认了三件非常重要的事：

1. `TB6612` 分支是一套新的 CubeMX/Keil 工程，目前还是硬件初始化骨架，应用层逻辑基本没接回来。
2. 旧 `DRV8874` 分支里已经有的 `gray_module / gray_monitor / motor_drive / oled_display / vision_protocol` 等模块，在 `TB6612` 分支里目前不存在。
3. 从 `.ioc` 与网表看，这块板更像是通过一个 7Pin 接口去控制外接 `TB6612` 模块，而不是板上直接焊了一颗 `TB6612` 芯片。

## 3. 本阶段确认下来的硬件关系

当前 `TB6612` 控制口可以整理为：

- `PA2` -> `DJ_PWM_A`
- `PA3` -> `DJ_AIN2`
- `PE9` -> `DJ_AIN1`
- `PE11` -> `DJ_STBY`
- `PE13` -> `DJ_BIN1`
- `PA15` -> `DJ_BIN2`
- `PE14` -> `DJ_PWM_B`

网表里对应出现了一个 `CN3` 七针接口，把上述控制线全部引出。  
这使当前主控方案的理解发生了一个很关键的收敛：

- 主控板负责输出控制信号
- 外接 `TB6612` 驱动模块负责真正带电机

## 4. 当前发现的问题

这轮调研里已经能提前看到两个会卡住后续开发的问题：

1. `PA15 / DJ_BIN2` 目前在工程里仍被配置成模拟输入，不是方向控制输出。
2. `TIM1 / TIM2` 的 PWM 周期仍是 CubeMX 默认大周期，暂时不适合作为正式电机 PWM 参数。

也就是说，当前这条分支虽然方向是对的，但还远没进入“可直接上车”的状态。

## 5. 对项目推进的意义

这轮扫描最大的意义是把后续路线理顺了：

- 不是从零重新做主控
- 而是把旧 `DRV8874` 分支已经跑通的上层逻辑迁移到新 `TB6612` 分支
- 再把电机底层改成 `TB6612` 驱动风格

这会比重新写一整套主控代码稳很多，也能最大化复用前面已经形成的调试成果。

## 6. 下一步最自然的动作

最自然的下一步已经很清楚：

1. 先把 `F103 -> F407` 的 `USART6` 短帧接收与 OLED 监视逻辑补回 `TB6612` 分支。
2. 再修正 `PA15` 与 PWM 基础参数。
3. 最后再写 `TB6612` 版电机驱动层。

这会继续保持当前项目一直在走的那条节奏：

- 先看见数据
- 再打通执行
- 最后做整车闭环

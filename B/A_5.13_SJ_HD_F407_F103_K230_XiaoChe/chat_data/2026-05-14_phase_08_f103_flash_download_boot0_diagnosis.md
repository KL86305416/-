# 阶段摘要 08：F103 下载失败与 BOOT0 异常定位
- 日期：2026-05-14
- 阶段编号：`08`
- 主题：`f103_flash_download_boot0_diagnosis`
- 范围：`F103 grayscale sensor(HD) / dev_logs / chat_data`

## 1. 本阶段触发背景

在 `F103` 运行态短帧已经完成并进入实板下载阶段后，用户遇到了新的阻塞：

- 编译成功
- 烧录过程显示 `Erase Done / Programming Done`
- 但随后出现大量 `Contents mismatch`
- 最终 `Flash Download failed - "Cortex-M3"`

这说明问题不在编译，而是在“写入后校验”这一层。

## 2. 本阶段先做了什么判断

本阶段先没有急着怀疑新加的协议代码，而是先从三层去分离问题：

1. 程序是否把 `SWD` 调坏了
2. `Keil` 工程下载配置是否不稳
3. 板级硬件状态是否异常

排查结果是：

- 当前代码没有关闭 `SWD`
- `HAL_MspInit()` 只是关闭 `JTAG`、保留 `SWD`
- 因此代码本身不像首要根因

## 3. 本阶段发现的两个关键线索

### 3.1 下载配置不够保守

工程目标器件是：

- `STM32F103C8`

但当前 `Keil` 调试配置里可见：

- `Flash algorithm` 使用的是 `STM32F10x_128.FLM`
- `SWD` 时钟为 `10MHz`

这意味着下载配置仍偏激进。

### 3.2 BOOT0 电平异常

用户随后补充了一个比软件更关键的现象：

- `BOOT0` 外接 `10K` 下拉到 `GND`
- 实测 `BOOT0` 仍有约 `1.7V`

结合网表进一步确认：

- 板上 `BOOT0` 实际还自带一颗 `R9 = 1k` 下拉到地
- 换句话说，这根线本应被很强地拉低

所以 `1.7V` 不是“普通漂浮”，而是明确异常。

## 4. 本阶段形成的核心结论

到这一阶段，问题优先级已经重新排序：

1. `BOOT0` 硬件电平异常
2. `Keil` 下载参数偏激进
3. 新写的应用逻辑

也就是说，目前更合理的判断是：

- 这次下载失败首先不像程序问题
- 更像板级异常，再叠加工程下载配置不够保守

## 5. 这对后续工作的意义

这个结论很重要，因为它把排障顺序重新拉正了：

- 现在不应该继续在应用代码里盲改
- 应先把 `BOOT0` 压回正常低电平
- 然后把 `SWD` 时钟和 `Flash algorithm` 收紧
- 再做二次下载验证

这一步相当于把“软件排障”及时拉回到了“硬件状态先校准”的正确路径上。

## 6. 下一步最自然的动作

本阶段之后，最自然的下一步是：

1. 仅保留最小系统，强制 `BOOT0 -> GND`
2. 重新测 `BOOT0` 是否接近 `0V`
3. 检查 `BOOT0` 到 `3V3` 或相邻引脚是否有异常耦合
4. 把 `Keil` 下载配置改成更保守参数
5. 再次尝试下载

只有这一步稳定以后，后面的 `F103 -> F407` 串口联调、运行态短帧验证和整车接入才有可靠基础。

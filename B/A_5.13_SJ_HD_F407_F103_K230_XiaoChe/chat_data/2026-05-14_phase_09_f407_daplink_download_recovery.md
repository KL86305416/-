# 阶段摘要 09：F407 DAP-Link 下载恢复分析
- 日期：2026-05-14
- 阶段编号：`09`
- 主题：`f407_daplink_download_recovery`
- 范围：`F407 main control panel / backups / dev_logs / chat_data`

## 1. 本阶段触发背景

在前面完成 `F407` 接收 `F103` 短帧、`DRV8874` 电机 PWM 基础层和 `OLED` 本地监视页之后，项目进入了实板下载阶段。  
这时用户补充了一个重要澄清：

- 当前出问题的不是 `F103`
- 而是 `F407` 主控
- 下载器不是 `ST-LINK`
- 而是 `DAP-Link`

报错形态也随之明确为：

- `CMSIS-DAP - Cortex-M Error`
- `RDDI-DAP Error`

## 2. 本阶段做出的关键判断

本阶段先把问题从“程序逻辑层”剥离出来，优先判断：

1. 是否是代码把 `SWD` 口搞坏了
2. 是否是 `DAP-Link/CMSIS-DAP` 配置不稳
3. 是否还叠加了工程时钟配置错误

结果是：

- 没发现代码重配 `PA13/PA14`
- 没发现关闭 `SWD/JTAG`
- 没发现看门狗、低功耗、读保护、选项字节之类的高风险动作

因此这次问题仍然不像业务代码本身导致。

## 3. 本阶段发现的两处核心问题

### 3.1 DAP-Link 实际连接时钟过高

虽然工程显示调试时钟像是 `1MHz`，但底层 `CMSIS_AGDI` 配置字符串实际仍然在跑：

- `10MHz`

对于 `DAP-Link` 来说，这一档位太激进，很容易直接造成：

- 握手失败
- 调试器附着失败
- `RDDI-DAP Error`

### 3.2 F407 的外部晶振常量写错了

本阶段还发现工程里有一处明确的板级配置错误：

- 软件 `HSE_VALUE = 25MHz`
- 实板原理图/网表里的晶振 `X1 = 8MHz`

这会让系统时钟推导、定时、串口波特率等一整串计算都处于错误基准上。

它未必单独构成下载失败的唯一原因，但它会让“上电后程序状态异常”的概率明显上升。

## 4. 本阶段已经落地的处理

这一轮没有停在口头分析，而是直接做了两处收敛：

1. 把 `CMSIS-DAP` 实际连接频率从 `10MHz` 改成 `1MHz`
2. 把 `F407` 的 `HSE_VALUE` 改成与实板一致的 `8MHz`

同时提前建立了备份目录，保证这轮修改可回溯。

## 5. 本阶段的意义

这一阶段的意义在于，它把“F407 下不进去程序”的问题重新定性为：

- 首先是下载链路和工程基础配置问题
- 不是当前新增寻迹/串口/OLED 业务代码直接把板子写坏

这让后续排障顺序变得更清晰：

- 先稳住 `DAP-Link`
- 再确保时钟参数正确
- 最后才去看是否还有硬件接线层面的问题

## 6. 下一步最自然的动作

本阶段之后，最自然的下一步就是：

1. 用收敛后的工程重新尝试连接和下载
2. 如果仍失败，再把 `SWD` 时钟降到更低
3. 使用 `Connect under Reset`
4. 核实 `VTref` 和 `NRST` 是否真的接入 `DAP-Link`
5. 必要时退回最小系统接线法验证

只有把这一关打通，后面 `F407` 主控读取 `F103` 数据并在 `OLED` 上稳定显示、再进入整车控制联调，才有可靠基础。

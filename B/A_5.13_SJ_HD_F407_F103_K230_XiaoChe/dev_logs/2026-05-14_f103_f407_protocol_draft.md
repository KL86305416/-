# F103 与 F407 报文定义草案

- 日期：2026-05-14
- 工作区：`D:\A_5.13_SJ_HD_F407_F103_K230_XiaoChe`
- 状态：`v0.1 draft`
- 目标：给主用灰度模块 `F103` 与主控板 `F407` 设计可快速落代码的 `I2C + UART` 通讯方案。

## 1. 设计结论

结合当前硬件结构，推荐如下：

- 主链路：`I2C`
- 备用/调试链路：`UART`

理由：

- `F407` 是主控，适合主动轮询 `F103`
- `F103` 输出的数据量很小，适合寄存器式读取
- `UART` 保留后可以方便调试、日志输出和链路备份

## 2. 物理层约定

### 2.1 主链路 I2C

| 项目 | 约定 |
| --- | --- |
| F407 侧 | `I2C3` |
| F103 侧 | `I2C2` |
| 连接口 | `F407 灰度-i2c-IN <-> F103 CN2` |
| 初始速率 | `100 kHz` |
| 推荐后续 | 联调稳定后可升到 `400 kHz` |
| 角色 | `F407 = Master`, `F103 = Slave` |
| 建议从地址 | `0x42` |

### 2.2 备用 UART

| 项目 | 约定 |
| --- | --- |
| F407 侧 | `USART6` |
| F103 侧 | `USART1` |
| 连接口 | `F407 灰度-串口-IN <-> F103 CN3` |
| 波特率 | `115200` |
| 格式 | `8N1` |

## 3. 数据语义约定

### 3.1 8 路灰度位定义

建议统一把 `F103` 的 8 路结果规范化为：

- `bit0 = 最左侧传感器 S1`
- `bit1 = S2`
- `bit2 = S3`
- `bit3 = S4`
- `bit4 = S5`
- `bit5 = S6`
- `bit6 = S7`
- `bit7 = 最右侧传感器 S8`

统一语义：

- `1 = 检测到有效黑线`
- `0 = 背景 / 无线`

说明：

- 即使硬件电平实际是反相的，也由 `F103` 固件内部做归一化
- `F407` 永远只看规范化结果，不关心具体电平高低

### 3.2 轨迹类型建议

| 值 | 含义 |
| --- | --- |
| `0x00` | `NONE`，无有效线 |
| `0x01` | `SINGLE_LINE`，单线 |
| `0x02` | `CROSS`，十字/大面积交叉 |
| `0x03` | `ALL_BLACK`，全黑 |
| `0x04` | `LEFT_EDGE`，左偏边缘 |
| `0x05` | `RIGHT_EDGE`，右偏边缘 |
| `0x06` | `AMBIGUOUS`，多峰不稳定 |

### 3.3 中心误差建议

`center_err` 定义为：

- 中心为 `0`
- 左偏为负
- 右偏为正
- 使用 `s16`

推荐缩放：

- 单位 `0.1 sensor pitch`

例如：

- 最左附近约 `-35`
- 中间约 `0`
- 最右附近约 `+35`

## 4. I2C 寄存器草案

### 4.1 只读状态块

建议 `F407` 每次从 `0x00` 开始连续读取 `10` 字节。

| 地址 | 名称 | 类型 | 说明 |
| --- | --- | --- | --- |
| `0x00` | `DEV_ID` | `u8` | 固定 `0xF1` |
| `0x01` | `PROTO_VER` | `u8` | 当前 `0x01` |
| `0x02` | `STATUS_FLAGS` | `u8` | 状态位 |
| `0x03` | `SENSOR_BITS` | `u8` | 8 路灰度结果 |
| `0x04` | `TRACK_TYPE` | `u8` | 轨迹类型 |
| `0x05` | `EDGE_COUNT` | `u8` | 边缘数，0/1/2/... |
| `0x06` | `CENTER_ERR_L` | `u8` | 中心误差低字节 |
| `0x07` | `CENTER_ERR_H` | `u8` | 中心误差高字节 |
| `0x08` | `FRAME_CNT_L` | `u8` | 帧计数低字节 |
| `0x09` | `FRAME_CNT_H` | `u8` | 帧计数高字节 |

### 4.2 `STATUS_FLAGS` 位定义

| 位 | 含义 |
| --- | --- |
| bit0 | 模块就绪 |
| bit1 | 数据有效 |
| bit2 | 传感异常 |
| bit3 | 全黑保护触发 |
| bit4 | 全白保护触发 |
| bit5 | UART 调试流已开启 |
| bit6 | 保留 |
| bit7 | 保留 |

### 4.3 可写控制块

| 地址 | 名称 | 类型 | 说明 |
| --- | --- | --- | --- |
| `0x10` | `CMD` | `u8` | 命令号 |
| `0x11` | `CMD_ARG0` | `u8` | 参数 0 |
| `0x12` | `CMD_ARG1` | `u8` | 参数 1 |
| `0x13` | `CMD_SEQ` | `u8` | 命令序号 |

### 4.4 建议命令集

| `CMD` | 名称 | 说明 |
| --- | --- | --- |
| `0x00` | `NOP` | 空操作 |
| `0x01` | `CLEAR_COUNTER` | 清帧计数 |
| `0x02` | `UART_STREAM_ON` | 开启 UART 状态流 |
| `0x03` | `UART_STREAM_OFF` | 关闭 UART 状态流 |
| `0x04` | `SOFT_RESET` | 软件复位 |

说明：

- 由于当前硬件阈值由电位器决定，不建议在协议里设计“软件调阈值”字段
- 协议只管读取状态和控制通信行为，不伪造不存在的硬件能力

## 5. I2C 轮询建议

### 5.1 推荐节奏

- 初期联调：`5 ms` 轮询一次，即 `200 Hz`
- 稳定后可试 `2 ms`，即 `500 Hz`

### 5.2 推荐流程

1. `F407` 写寄存器起始地址 `0x00`
2. `F407` 重启后连续读 `10` 字节
3. 解析：
   - `sensor_bits`
   - `track_type`
   - `center_err`
   - `frame_cnt`

## 6. UART 备用协议草案

建议 `F103` 的 UART 复用和 `OpenMV` 一样的串口帧封装：

| 字段 | 字节数 | 说明 |
| --- | --- | --- |
| `SOF1` | 1 | `0xAA` |
| `SOF2` | 1 | `0x55` |
| `VER` | 1 | `0x01` |
| `SRC` | 1 | `0x20 = F103` |
| `TYPE` | 1 | 消息类型 |
| `SEQ` | 1 | 帧序号 |
| `LEN` | 2 | payload 长度 |
| `PAYLOAD` | `LEN` | 负载 |
| `CRC16` | 2 | `CRC16-CCITT-FALSE` |

### 6.1 消息类型

| `TYPE` | 名称 | 说明 |
| --- | --- | --- |
| `0x10` | `GRAY_STATUS` | 主状态帧 |
| `0x11` | `GRAY_DEBUG` | 调试帧 |
| `0x12` | `GRAY_HEARTBEAT` | 心跳 |
| `0x7F` | `ACK` | 应答 |

### 6.2 `GRAY_STATUS` 负载

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `status_flags` | `u8` | 状态位 |
| `sensor_bits` | `u8` | 8 路灰度结果 |
| `track_type` | `u8` | 轨迹类型 |
| `edge_count` | `u8` | 边缘数 |
| `center_err` | `s16` | 中心误差 |
| `frame_cnt` | `u16` | 帧计数 |

长度共 `8` 字节，够小，也便于 F407 直接映射成结构体。

### 6.3 UART 发送节奏

建议：

- 正式运行默认关闭持续流
- 仅在调试期，或收到 `UART_STREAM_ON` 后才每 `10 ms` 发一次

这样能避免串口长期占用不必要带宽。

## 7. F407 侧推荐处理逻辑

### 7.1 I2C 主路径

`F407` 每个控制周期：

- 轮询一次 `F103`
- 检查 `DEV_ID`
- 读取 `sensor_bits`
- 读取 `track_type`
- 读取 `center_err`

### 7.2 UART 备用路径

`F407` 可在以下场景切换或同时打开 UART：

- 联调期抓日志
- I2C 数据异常时做交叉验证
- 需要对 `F103` 输出做串口抓包分析

## 8. 当前推荐落地顺序

1. 先实现 `I2C` 固定状态块读取
2. 再实现 `F407` 侧的灰度结构体解析
3. 最后补 `UART` 调试流

## 9. 推荐的最小结构体

### 9.1 F103 侧内部输出结构

```c
typedef struct {
    uint8_t status_flags;
    uint8_t sensor_bits;
    uint8_t track_type;
    uint8_t edge_count;
    int16_t center_err;
    uint16_t frame_cnt;
} GrayStatusFrame;
```

### 9.2 F407 侧缓存结构

```c
typedef struct {
    uint8_t online;
    uint8_t sensor_bits;
    uint8_t track_type;
    int16_t center_err;
    uint16_t frame_cnt;
    uint32_t last_update_ms;
} GrayModuleState;
```

## 10. 当前最稳的工程建议

- `I2C` 作为主链路先跑通
- `UART` 作为调试和备用链路保留
- 不要在第一版协议里加入阈值软件配置，因为当前硬件阈值本来就是电位器调的
- 先把 `sensor_bits + track_type + center_err` 这 3 个最关键字段打通

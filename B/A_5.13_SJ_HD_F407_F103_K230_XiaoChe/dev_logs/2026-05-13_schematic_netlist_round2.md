- 目的：把源码分析与 `.enet` 网表分析合并，形成一版更可靠的硬件理解。

## 1. 文件性质确认

本轮确认了两类文件的性质：

- `.enet` 是可直接解析的 JSON 网表
- `.epro2` 是工程压缩包，内含 `project2.json` 和更大的工程主体文件

因此本轮核心结论主要以 `.enet` 为准，因为它已经足够提供：

- 器件清单
- 引脚网络
- 接口去向

## 2. 本轮已确认的设计决策

以下 3 点现已确认：

- `F103` 是主用灰度模块
- `F407` 的 `灰度-IN` 对应另一块无 MCU 的 8 路灰度板
- 视觉侧最终接 `OpenMV`，`K230` 作为备选

## 3. F103 的硬件结论

### 3.1 已经坐实的器件结构

`F103` 网表直接给出了完整器件关系：

- `U1 ~ U8 = ITR20001/T`
- `U9 / U10 = LM324DR2G`
- `R1 ~ R8 = JER655HC-103`
- `U11 = STM32F103C8T6`

所以这块板的本质不是“简单传感器板”，而是：

- 8 路灰度采集
- 8 路门限可调
- 运放比较/整形
- MCU 二次处理

### 3.2 8 路灰度通道映射

网表已经坐实以下关系：

- `IN1 -> LM324 -> PA2`
- `IN2 -> LM324 -> PA3`
- `IN3 -> LM324 -> PA4`
- `IN4 -> LM324 -> PA5`
- `IN5 -> LM324 -> PB12`
- `IN6 -> LM324 -> PB13`
- `IN7 -> LM324 -> PB15`
- `IN8 -> LM324 -> PA8`

说明：

- `F103` 最终拿到的是整形后的 8 路灰度结果
- 它不是把 8 路原始信号直接裸传给 `F407`

### 3.3 阈值调节已坐实

`R1 ~ R8` 都是三脚可调电阻，连接方式统一为：

- 一端 `3V3`
- 一端 `GND`
- 中间滑动端进 `LM324` 正输入

这说明：

- 每一路灰度门限都可独立调节
- `F103` 是一块可调阈值的智能灰度模块

### 3.4 F103 对外接口已坐实

`CN2`：

- `3V3 / GND / PB10 / PB11`
- 对应 `I2C2`

`CN3`：

- `3V3 / GND / OUT_RX / OUT_TX`

其中一个关键细节是：

- `OUT_RX` 实际接的是 `F103 USART1_TX`
- `OUT_TX` 实际接的是 `F103 USART1_RX`

也就是说，这组命名是按“对端设备视角”命名的，不是按 MCU 自己视角命名的。

## 4. F407 的硬件结论

### 4.1 主电源链路已坐实

主电源结构为：

- `XT30` 电源输入
- 保险丝
- 电源开关
- `12V` 母线
- DCDC 转 `3.3V`

所以：

- 主板主供电来自 `12V`
- USB Type-C 不是主供电入口

### 4.2 灰度链路有两种接法

第一种：

- `灰度-IN`
- 8 路原始灰度输入直接进 `F407`
- 现已确认对应另一块无 MCU 灰度板

第二种：

- `灰度-i2c-IN`
- `灰度-串口-IN`
- 现已确认是为 `F103` 这类智能灰度模块准备的

### 4.3 F103 与 F407 的对接关系

当前已经可以明确写成结论：

- `F103` 应接 `F407` 的 `灰度-i2c-IN` 和/或 `灰度-串口-IN`
- `F103` 不对应 `F407` 的 `灰度-IN`
- `灰度-IN` 是留给另一块无 MCU 的 8 路灰度板的

而且 `F103` 与 `F407` 的 I2C/UART 接口在脚位顺序上是高度匹配的，这不是偶然。

## 5. 视觉链路的确认结果

### 5.1 F407 给视觉侧的接口

`视觉供电out`：

- `GND + 12V`

`视觉信号out`：

- `UART4`

### 5.2 当前确认的设计意图

- 视觉主路线：`OpenMV`
- 视觉备选：`K230`

因此这套接口更应理解为：

- 给 `OpenMV` 预留的主用供电与串口链路
- 同时兼容未来切换到 `K230`

## 6. 其它已坐实的模块

### 6.1 姿态模块

网表里存在独立的姿态器件 `U4 = JYXXX 陀螺仪`，并且它同时连到：

- `I2C2`
- `USART3`

这说明姿态模块不是猜测项，而是硬件真实存在的子模块。

### 6.2 显示模块

板上并存两种显示方案：

- I2C OLED
- SPI OLED

这不是命名混乱，而是硬件方案确实双轨并存。

### 6.3 存储模块

TF / MicroSD 与 `SPI1`、`CS`、`DET` 的连接关系被网表坐实。

### 6.4 执行器与反馈

当前最合理的理解是：

- `TIM1 + TIM2`：驱动输出
- `TIM12`：两路舵机 PWM
- `TIM3 + TIM4`：两组编码器反馈

### 6.5 CAN

当前网表只说明：

- MCU 的 CAN 引脚被直接引到接口

还没有看到板载 CAN 收发器，所以它暂时更像：

- CAN 信号引出
- 调试或后续扩展接口

## 7. 对上一轮结论的修正

上一轮里有几条推断，现在可以升级或修正：

- `F103` 不是“只有简单 IO 测试固件的小板”，而是硬件上完整的主用灰度模块
- `F103` 与 `F407` 的关系不是“可能配套”，而是按配套方向设计的
- `F407` 的 `灰度-IN` 不是给 `F103`，而是给另一块无 MCU 灰度板
- 视觉主路线现在应明确写成 `OpenMV`，`K230` 为备选

## 8. 当前最稳的一版系统理解

一句话概括：

- `F103` 负责灰度采集与本地整形处理
- `F407` 负责整车主控与多模块集成
- `OpenMV` 是视觉主路线
- `F407` 还兼容一块无 MCU 的直接灰度板

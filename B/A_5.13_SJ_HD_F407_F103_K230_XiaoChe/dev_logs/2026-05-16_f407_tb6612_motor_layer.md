## 1. 触发背景

在前一阶段修复 `USART6` 接收中断链后，`TB6612` 分支已经能够正常解析来自 `F103` 灰度板的运行态短帧。

这意味着主控当前的主要短板不再是“看不到灰度数据”，而是“电机输出层尚未真正建立”。

因此这次工作的目标很明确：

1. 把 `TB6612` 电机控制底层从 CubeMX 骨架状态落到可用状态
2. 为后续寻迹控制逻辑预留稳定、统一的驱动接口
3. 保证上电默认安全，不出现一下载就跑车的风险

## 2. 本轮前确认出的两个底层硬伤

本轮正式下手前，确认 `TB6612` 分支仍有两个会直接卡住电机控制的硬问题：

1. `DJ_BIN2` 对应的 `PA15` 仍被配置成 `GPIO_Analog`
2. `TIM1 / TIM2` 的 PWM 周期仍是 CubeMX 默认大周期，不适合直接做电机 PWM

如果不先修这两个点，即使上层已经开始写 `TB6612` 控制逻辑，也无法形成稳定、可预期的双电机输出。

## 3. 本次新增的电机驱动抽象

本轮新增：

- `Core\Inc\motor_drive.h`
- `Core\Src\motor_drive.c`

这套接口不是临时点机代码，而是后续主控逻辑可以继续复用的底层抽象。

当前提供的能力：

- `MotorDrive_Init()`
- `MotorDrive_SetStandby()`
- `MotorDrive_StopAll()`
- `MotorDrive_BrakeAll()`
- `MotorDrive_SetPercent()`

### 3.1 当前抽象的控制对象

当前按 `TB6612` 原生结构管理两路电机：

- `MOTOR_DRIVE_CHANNEL_A`
- `MOTOR_DRIVE_CHANNEL_B`

暂时没有直接写成“左电机/右电机”，是因为目前仅能从原理图和网表确认 `A/B` 通道定义，尚未最终确认它们在整车上的左右实际安装方向。

这样做的好处是：

- 底层先保持电气定义准确
- 后续上车后再把 `A/B` 映射成“左/右”即可

### 3.2 当前的方向与占空比策略

当前 `MotorDrive_SetPercent()` 采用：

- 正值：正转
- 负值：反转
- `0`：空转停机（coast）

另外单独保留：

- `MotorDrive_BrakeAll()`：双电机短刹

这样上层后面无论是手动测试、串口调试还是寻迹控制，都可以直接用“有符号百分比”作为统一控制输入。

## 4. 这次同步修掉的底层配置

### 4.1 GPIO

`PA15 / DJ_BIN2` 已从模拟输入改为普通推挽输出，和 `DJ_AIN1 / DJ_AIN2 / DJ_BIN1 / DJ_STBY` 一起组成完整的 `TB6612` 方向控制集合。

### 4.2 PWM 参数

为了让 `TIM1` 和 `TIM2` 都输出适合电机驱动的 PWM，本轮统一调整为约 `20kHz`：

- `TIM1`：`Prescaler = 7`，`Period = 1049`
- `TIM2`：`Prescaler = 3`，`Period = 1049`

这样做的好处有两个：

1. `TIM1` 和 `TIM2` 采用同样的自动重装值，后续双电机占空比控制尺度一致
2. PWM 频率进入更适合电机驱动的区间，避免继续使用默认超大周期

### 4.3 工程同步

本轮不仅修改了源代码，还同步更新了：

- `5.15_F407_TB6612.ioc`
- `MDK-ARM\5.15_F407_TB6612.uvprojx`

这意味着：

- 后续重新打开 CubeMX / Keil 时，不会出现“代码改了但工程文件没跟上”的脱节

## 5. `main.c` 当前变化

`main.c` 已接入：

- `#include "motor_drive.h"`
- `MotorDrive_Init()`

当前上电初始化顺序里，电机层会在主循环业务前先完成底层启动。

同时，`MotorDrive_Init()` 内部采用的是“先拉低 `STBY`，启动 PWM 通道并清零，再打开 `STBY`”的顺序，目的是保证电机驱动上电过程尽量平稳，不在初始化瞬间输出异常脉冲。

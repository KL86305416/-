## 4. 代码现状

### 4.1 当前 `main.c` 状态

`TB6612` 分支的 `main.c` 仍是典型 CubeMX 生成骨架：

- 完成 `GPIO / I2C / SPI / TIM / UART / USB` 初始化
- `while (1)` 主循环为空
- 没有任何用户业务层初始化
- 没有运行态轮询逻辑

这意味着它目前还没有接入：

- `F103` 运行态短帧接收
- `OLED` 数据监视
- 电机驱动抽象层
- OpenMV 或视觉协议层
- 按键/指示灯联调逻辑

### 4.2 与 DRV8874 分支的差异

旧的 `DRV8874` 分支中已经存在这些上层模块：

- `gray_module`
- `gray_monitor`
- `motor_drive`
- `oled_display`
- `vision_protocol`
- `can`

而 `TB6612` 分支当前只有 CubeMX 默认文件，没有这些业务模块。  
所以当前最准确的判断是：

- `DRV8874` 分支：已经有“应用层雏形”
- `TB6612` 分支：目前只有“硬件工程骨架”

## 5. IOC 侧硬件映射

### 5.1 MCU 与基础工程信息

从 `.ioc` 可确认：

- MCU：`STM32F407VETx`
- 封装：`LQFP100`
- 外部高速时钟：`HSE`
- 时钟目标：`168MHz`

这与当前主控板所使用的 `STM32F407VE` 方向一致。

### 5.2 电机控制相关引脚

当前 `.ioc` 与 `main.h` 对 `TB6612` 控制逻辑的映射如下：

- `PA2` -> `DJ_PWM_A` -> `TIM2_CH3`
- `PA3` -> `DJ_AIN2` -> GPIO 输出
- `PE9` -> `DJ_AIN1` -> GPIO 输出
- `PE11` -> `DJ_STBY` -> GPIO 输出
- `PE13` -> `DJ_BIN1` -> GPIO 输出
- `PA15` -> `DJ_BIN2` -> 当前竟然是 `GPIO_Analog`
- `PE14` -> `DJ_PWM_B` -> `TIM1_CH4`

这组定义非常符合 `TB6612` 双电机驱动控制方式：

- A 电机：`PWMA + AIN1 + AIN2`
- B 电机：`PWMB + BIN1 + BIN2`
- 全局待机：`STBY`

### 5.3 现阶段两个关键异常点

当前这套工程里，电机驱动部分至少有两个明确未完成点：

1. `DJ_BIN2` 对应的 `PA15` 仍被配置成模拟输入，不是 GPIO 输出。
2. `TIM1` 与 `TIM2` 的 PWM 周期仍处于 CubeMX 默认大周期状态。

当前定时器参数为：

- `TIM1.Period = 65535`
- `TIM2.Period = 4294967295`

这不适合作为最终电机 PWM 频率配置，说明电机控制部分还没有真正调到可上车的状态。

## 6. 其他通信与外设资源

这套 `TB6612` 主控分支保留了与旧主控方案高度一致的通信框架：

- `UART4`：`PA0/PA1`，对应 `OpenMV`
- `UART5`：`PC12/PD2`，当前为预留串口
- `USART3`：`PD8/PD9`，对应 `YJ901S`
- `USART6`：`PC6/PC7`，对应灰度串口，也就是后续 `F103 -> F407` 串口链路
- `I2C1`：`PB6/PB7`，对应 I2C OLED
- `I2C2`：`PB10/PB11`，对应 `YJ901S` I2C
- `I2C3`：`PA8/PC9`，对应灰度 I2C
- `SPI1`：TF 卡
- `SPI2`：SPI OLED

辅助 IO 也都还在：

- `PD3~PD6`：`KEY1~KEY4`
- `PB9 / PE0`：`LED1 / LED2`
- `PB14 / PB15`：两路舵机 PWM
- `TIM3 / TIM4`：两路编码器
- `PE2~PE6 + PC13~PC15`：原始 8 路无 MCU 灰度输入

说明这块板子的总体架构并没有被推翻，只是“电机驱动实现方案”从旧分支切到了 `TB6612` 控制方式。

## 7. 网表/原理图关系分析

### 7.1 CN3 很关键

网表中最关键的新发现是 `CN3`：

- `CN3` 是一个 `1x7P` XH 接口
- 7 个引脚分别接到：
  - Pin1 -> `A2`
  - Pin2 -> `A3`
  - Pin3 -> `DJ-AIN1`
  - Pin4 -> `DJ-STBY`
  - Pin5 -> `DJ-BIN1`
  - Pin6 -> `A15`
  - Pin7 -> `DJ-PWM-B`

再结合 `.ioc` 中的功能命名，可以把这 7 根线解释为：

- `A2` = `PA2` = `DJ_PWM_A`
- `A3` = `PA3` = `DJ_AIN2`
- `DJ-AIN1` = `PE9`
- `DJ-STBY` = `PE11`
- `DJ-BIN1` = `PE13`
- `A15` = `PA15` = `DJ_BIN2`
- `DJ-PWM-B` = `PE14`

也就是说，`CN3` 本质上就是一组完整的 `TB6612` 外接控制口。

### 7.2 没有发现板载 TB6612 芯片

本轮在 `.enet` 文本网表中，没有找到明确的：

- `TB6612`
- `AIN1/AIN2/BIN1/BIN2/PWMA/PWMB/STBY` 对应的板载驱动芯片实体
- `AO1/AO2/BO1/BO2` 这类典型电机输出网络

因此当前更合理的判断是：

- 这块主控板并未把 `TB6612` 直接做成板载芯片
- 而是通过 `CN3` 把控制信号引出，去连接一块外部 `TB6612` 电机驱动模块

这个判断与您现在“采用 TB6612 驱动”的说法并不冲突，反而更吻合实际接线方式。

### 7.3 电源与辅助模块

从网表还能确认几类板上模块：

- `CN1`：`XT30` 电源输入
- `U3`：`20*28-DCDC`，把 `12V` 电源降到 `3.3V`
- `U4`：`JY901S` 模块接口
- `U5`：4 针 `I2C OLED`
- `PWM.1 / PWM.2`：两路 PWM 外接接口，对应 `PB14 / PB15`
- `视觉供电out`：一组 `GND + 12V` 输出接口

这说明主控板本身承担的是：

- 供电分配
- 主控运算
- 传感器接入
- 外部执行器/驱动模块控制

## 8. 当前最重要的工程判断

如果我们后面要推进 `F407` 主控代码，当前最稳妥的理解应该是：

1. 上层控制逻辑仍应沿用之前已经验证过的 `DRV8874` 分支思路。
2. 但电机输出后端必须改写为 `TB6612` 风格：
   - A 电机：`PWMA + AIN1 + AIN2`
   - B 电机：`PWMB + BIN1 + BIN2`
   - 额外处理 `STBY`
3. 真正下手改代码前，还需要先把 `PA15` 和 PWM 频率这两个底层问题修正。

换句话说，下一阶段不是“重新设计整套主控架构”，而是：

- 把 `DRV8874` 分支已经形成的上层能力迁过来
- 再把电机底层适配成 `TB6612`

## 9. 风险点

当前分支存在的主要风险如下：

1. `PA15 / DJ_BIN2` 仍为模拟模式，电机 B 路方向控制不完整。
2. `TIM1/TIM2` PWM 周期未收敛到合适频率，电机驱动参数不具备直接实车使用条件。
3. `USART6` 虽然硬件资源已经在，但接收逻辑尚未接入，当前 `F103 -> F407` 链路在这套工程里实际上还没跑起来。
4. `TB6612` 若为外接模块，则必须在代码和接线文档中同时明确 `CN3` 的实际线序，避免后面接反。
5. 当前没有板载 `TB6612` 芯片文本证据，因此后续实现应默认按“外接驱动模块”处理，除非原理图图形页面再给出相反证据。

## 10. 建议的下一步

建议的推进顺序如下：

1. 先在 `TB6612` 分支中补回旧分支已经验证过的 `USART6` 短帧接收与 OLED 监视能力。
2. 再修正 `PA15` 为 GPIO 输出，并把 `TIM1/TIM2` 调到明确可用的电机 PWM 频率。
3. 然后新建 `TB6612` 版 `motor_drive` 层，专门处理 `PWMA/PWMB + AINx/BINx + STBY`。
4. 最后才进入主控寻迹输出与实车联调。

这样做的好处是：

- 先把“看得到数据”建立起来
- 再做“驱得动电机”
- 最后再做“跑得对路线”

这个顺序最稳，也最符合当前项目已经形成的开发节奏。

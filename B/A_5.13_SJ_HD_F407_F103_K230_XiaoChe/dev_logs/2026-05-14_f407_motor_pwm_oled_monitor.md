## 3. 电机 PWM 本次实际收正内容

### 3.1 驱动模型

本次代码明确按当前实际使用的：
- `DRV8874`

来建模，也就是：
- `4` 路电机
- 每路 `IN1 + IN2`
- 双 PWM 驱动一台电机

### 3.2 新建 `motor_drive` 层

新增 `motor_drive` 模块后，主控侧已经具备：
- `MotorDrive_Init()`
- `MotorDrive_StopAll()`
- `MotorDrive_SetPercent()`

当前 `MotorDrive_SetPercent()` 使用：
- 有符号百分比
- `+` 表示一方向
- `-` 表示反方向
- `0` 表示双路都关断

这样后面把 `TURN`、差速控制或姿态闭环接进来时，接口会更干净。

### 3.3 PWM 基础参数

本次把电机相关两个关键定时器的周期先统一到了 `20kHz`：

- `TIM1`
  - 计数时钟约 `168 MHz`
  - `ARR = 8399`
  - PWM 频率约 `20 kHz`

- `TIM2`
  - 计数时钟约 `84 MHz`
  - `ARR = 4199`
  - PWM 频率约 `20 kHz`

这比之前 `TIM2 = 0xFFFFFFFF` 的状态要合理得多，也终于进入了“可作为电机 PWM 基础”的状态。

### 3.4 当前安全策略

虽然 PWM 层已经收正，但本次仍然保持：
- 初始化后默认全部电机输出为 `0`

也就是说：
- 本次不会因为新代码接入，就让电机在上电后自行动作

这一步是刻意保守处理的。

## 4. OLED 监视页本次实际接入内容

### 4.1 选用的显示路径

本次优先选择的是：
- `U5`
- `I2C1`
- `PB7 / PB8`

原因很直接：
- 板上已经有明确的 `I2C OLED` 预留
- 当前代码里没有现成显示库
- 用 `I2C` 起一版最小监视页，比从 `SPI OLED` 起步更快

### 4.2 当前显示驱动假设

这次接入时采用的是一个明确但可调整的假设：

- 面板是 `SSD1306` 兼容类 `128x64`
- `I2C` 地址优先尝试 `0x3C`
- 次选尝试 `0x3D`

这意味着：
- 如果板上实际是同类 `SH1106/SSD1306` 派生屏，大概率可以很快点亮
- 如果屏型号或地址不同，也已经有一条明确的最小改动路径去调整

### 4.3 新建显示层

新增 `oled_display` 模块，提供：
- 初始化
- 清屏
- 按行写字符串
- 整屏刷新

新增 `gray_monitor` 模块，负责把 `GrayModuleState` 中的运行态数据格式化成 OLED 页面。

## 5. 当前 OLED 显示的内容

当前监视页每 `200 ms` 更新一次，主要显示：

1. `F103 GRAY RX`
2. `Online / Age`
3. `Turn`
4. `Scene`
5. `Flags`
6. `LINE_FOUND / CENTER_ON_LINE`
7. `LEFT_FEATURE / RIGHT_FEATURE / WIDE_BLACK`
8. `UNSTABLE / SENSOR_FAULT / VALID`

换句话说，现在你在主控板上就能直接看到：
- `F103` 有没有在线
- 数据是不是持续在更新
- 当前建议左转还是右转
- 当前识别成了什么场景
- 各状态位是否在跳变

4. 通过移动灰度板或遮挡线路，观察 OLED 上数据是否同步变化

## 10. 若 OLED 没亮，最先检查什么

如果上板后 OLED 没亮，优先检查这几件事：

1. 板上当前实际焊接的是 `U5 I2C OLED` 还是 `U6 SPI OLED`
2. `U5` 是否真的是 `SSD1306` 兼容类
3. 地址是否为 `0x3C / 0x3D`
4. 是否存在列偏移或 `SH1106` 类兼容差异

也就是说，当前代码已经把“第一条能跑起来的路径”搭好了，如果屏参数有偏差，调整面会比较小。
